A Video Object Notation, by Mark Filipak (markfilipak.contact+notation@gmail.com)

Thank you for reading and commenting.

Synopsis.
The presentation introduces a notation that can model video streams to support planning (and optionally, implementation) of remuxing and transcoding tasks.
Stream modeling (and optionally, stream processing) is object oriented, element by element, not via filter functions. Stream editing is via primatives, each of limited scope. Pixel editing is via methods attached to element tokens.
Streams are traversed by strides of elements. The various types of strides and their properties are each characteristic of the various video formats.

Advantages for Users:
When you write a notation you're specifying what you have and what you want, not what filters you think might work.
No more guessing or trial and error. You are in control.

Advantages for Developers:
Users directly create and manage streams, so you don't have to.
The required stream primitives are straightforward, easy to write, and easy to test.
When a primitive works, bug-free, it's done, forever.
Since users don't select primitives and may not even know they exist, user documentation is not needed.
No more modifying stream filters to accommodate new use cases.
No more duplicating code snippets to extend use cases across various disparate stream filters.
