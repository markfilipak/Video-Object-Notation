Synopsis

Commonly Found

Comments, Whitespace, Errors, Abbreviations
Stripped from notations during parsing — stripped in the order shown.

1, Comments: // (2 backslashes) & all text following them to the end-of-line are stripped.

2, Whitespace: All spaces & end-of-line codes are stripped. So, spaces & line breaks can be freely used to format notations.

3, Supplementary stream rate identifiers: =rateftfps =ratehpps =ratehtfps =ratepfps =ratepps =ratesfps =ratespfps =ratespps =ratesps =ratestfps =ratetpfps =ratetpps, are stripped. They can be freely used anywhere to enhance comprehension.


Errors.

? YYYY-MM-DD hhmmss line# good_text Bad text: bad_text

 • There is no error indicator other than '?'. There is no use of '?' as a 'printed' character other than as an error indicator. And '?' is always the first character of an error line.

 • YYYY-MM-DD hhmmss is the wall clock time of the error.

 • good-text is the full text of line# that preceded the error.

 • bad-text is the property or method that provoked the error.

 • Any remaining notation is clipped off.


Abbreviations.
"'0.96" means "1/0.96", "'1.001" means "1000/1001", etc. Abbreviations make true rates useable in file names (which has been found to be very handy).

System Tokens & Operators
_  Underline is used to separate notational elements that might otherwise be confusing, for example, pps_hpps.
$  Dollar sign by itself is the current stream. It is rarely needed.
$  Dollar sign followed by a name is a temporary variable. For example, $name="notation".
&  Ampersand is an overload operator. It must be declared before use. For example,
&="blend"(AB)(A&B) transforms two elements, 'A' & 'B', to make one element, 'A&B', that blends 'A' & 'B'.
&="interpolate"(AC)(A&C) transforms two elements, 'A' & 'C', to make one element, 'A&C', that conceptually is 'B'.
#  Number sign — think: a number of times — identifies a looper. Example: (5#notation), runs notation 5 times.
? :  From 'C': (test?notation_if_true:notation_if_false), (test?notation_if_true:), (test?:notation_if_false). Also accepted: (test?notation_if_true).
.. 

Two dots denotes a stride-match expansion. For example:

 • A..Z is equivalent to ABCDEFGHIJKLMNOPQRSTUVWXYZ.

 • Aa..Zz is equivalent to AaBbCcDdEeFfGgHhIiJjKkLlMmNnOoPpQqRrSsTtUuVvWwXxYyZz.

 • Ab..Yz is equivalent to AbCdEfGhIjKlMnOpQrStUvWxYz.

 • Aa..LlLm..WxXx is equivalent to AaBbCcDdEeFfGgHhIiJjKkLlLmMnNoOpPqQrRsStTuUvVwWxXx.

Audio & Subtitle Streams

.a.property_or_method operates on all audio streams — this is usually what you want.

.an.property_or_method is notation that focuses on audio stream 0 1 2...

.an returns '?' if audio stream n does not exist.

.s.property_or_method operates on all subtitle streams — this is usually what you want.

.sn.property_or_method is notation that focuses on subtitle stream 0 1 2...

.sn returns '?' if subtitle stream n does not exist.


Audio synchronization.
Audio resampling is needed to synchronize audio during forced frame rate changes. It can be done by remuxing, e.g., 24'1.001fps_24fps.a.resample("1.001").

Example: When mastering cinema to cine-slow, running time stretches by 1001/1000. Resampling audio by 1.001 (which makes more samples) stretches audio to fit the video (which unfortunately makes tone a little flat).
Compare 24pps
to 24pps]24'1.001fps.a.resample("1.001")
(Can't hear the difference? Try quickly clicking both.)
When remuxing back to cinema, 24'1.001fps_24fps.a.resample("'1.001") reverses the above.

Example: When mastering cinema to PAL-fast, running time shrinks by 24/25. Resampling audio by 0.96 (which makes fewer samples) shrinks audio to fit the video (which unfortunately makes tone a little sharp).
Compare 24pps
      to 24pps]25fps.a.resample("'0.96")
When remuxing back to cinema, 25fps_24fps.a.resample("0.96") reverses the above.

Any factor is possible. 1.001 (or 1001/1000) & '1.001 (or 1/1.001 or 1000/1001) & '0.96 (or 1/0.96 or 25/24) & 0.96 (or 24/25) are just the most common factors.

Subtitle synchronization.
fps.s.resample(factor) essentially works similarly for subtitltes.

Note how the stride-matches have been constructed to look for the latest possible pattern, first. That usually produces the best and fastest results.

Also note that there are 12 nested stride-matches. One of the best properties of nested stride-matches is that once a nesting level finds the right pattern, it loops at that level (without traversing the hierarchy) until the pattern fails. When the pattern fails, that nesting level's miss is executed. Within that miss term are found the inner nesting levels.

Eventually, all inner levels miss. At that time, the hierarchy is traversed once more from the beginning looking for the current 50 halfpics against the patterns. If all levels miss, then reverse 24-to-25 telecine finishes with status: all_miss.

When reverse 24-to-25 telecine finishes with all_miss, it abandons the current 50 halfpics and passes just the first 2 halfpics to the output and increments the next pointer by 2 instead of by 50. It then resumes looking for 12, 50 halfpic patterns in the remaining input stream until the input stream is exhausted.

The preceding description doesn't apply solely to reverse 24-to-25 telecine. It applies to all nested stride-matches. Building an equivalent nested, back-&-forth, FFmpeg filter complex would be insanely difficult. In fact, building an equivalent FFmpeg filter complex is probably impossible. The above notation accomplishes it in just 390 characters:

Below is standard 8-to-10 telecine followed by 2 smooth video alternatives. They illustrate how the notation can make complex operations compact and easy to understand.

8-to-10 Telecine Via 2-3-2-3 Pull-Down 
//  1/240 sec-per-char +-all-strides-------------------------->
24pps               // (Aa______)(Bb______)(Cc______)(Dd______) source pps
_hpps(Aa..Dd)       // (A_______)(B_______)(C_______)(D_______) source top hpps
                    // (a_______)(b_______)(c_______)(d_______)        bottom hpps
(AaBbBcCdDd)=60hpps // (A_-20%)(B_________+60%)(C_-20%)(D_-20%) target top hpps
                    // (a_-20%)(b_-20%)(c_-20%)(d_________+60%)        bottom hpps
]30'1.001htfps      // [A+a___][B+b___][B+c___][C+d___][D+d___] target htfps
                    //                 ================         combed
PTSs: Forced by 1/1.001 — Combing: 40% — Judder: 53% [see note 1] @ 6 jps (judders−per−second) [see note 2] — Cadence: 4+3+3+3+3 [see note 3] — Burden: None — Generations: 0-0-0-0-0.

[note 1] Judder is the worst case 'jump' or 'pause' at a picture-to-picture or frame-to-frame change.
For top halfpics,

Consider these: (A_______)(B_______),
  versus these: (A_-20%)(B_________+60%). 		     The time difference is -2/10 characters = -20% followed by +6/10 characters = +60%. The judder is the absolute sum = 80%.
By identical consideration of bottom halfpics (c_-20%)(d_________+60%), the judder is also 80%. The 2 judders overlap by only 1/3rd, so when the top & bottom halfpics are superimposed, the overall judder is reduced to 53%.

[note 2] One judder per 1/6 second = 1/(1/6) jps.

[note 3] Cadence is smoothness across entire strides. To visualize it, imagine a clip showing a mouse running across a tiled floor.
For top halfpics, during 5 uniform frame times, the mouse appears to traverse 2+1+1+2+2 'tiles'.
For bottom halfpics, during 5 uniform frame times, the mouse appears to traverse 2+2+2+1+1 'tiles'.
Superimposing the top & bottom halfpics, the mouse appears to traverse 4/2+3/2+3/2+3/2+3/2 'tiles' = 4+3+3+3+3 as integers.

[note 1] The interpolated pictures are perfectly smooth but they're not at a constant rate whereas the 30'1.001pfps frames are forced to a constant rate, so they judder. The largest difference (largest judder) is between I1 & I2.

[note 3] Except for the end stride, during 5 uniform frame times, the mouse appears to traverse 15+15+20+15+15 'tiles' (which reduces to 3+3+4+3+3).

[note 4] Interpolations = 0+2+1+1+2 per 1/6 video second = 6/(1/6).


Distinguishing between images versus frames is useful. To appreciate the difference consider this: 4-to-5 interpolation creates strides of interpolated pictures having durations 1/32, 1/32, 1/24, 1/32, 1/32 seconds with zero judder and no cadence. But forcing the pictures into 1/30 second frames produces 27% judder @ 6 jps & 3+3+4+3+3 cadence.

An Actual Case
What follows is a 4-step example of how to extend a basic notation to expand coverage from one stream to multiple streams. It addresses the integration and improvement of a so-called "Making Of" documentary that's a mix of 'NTSC'-hard segments & NTSC-scan segments (with a twist at the end).

Step 1. The 1st notation makes pictures of the 'NTSC'-hard segments via 10-to-8 detelecine.

30'1.001htfps[hpps      // decode
(AaBbBcCdDd)            // stride-1: ..1.1..2.2
(                       //
 (                      //
  (Aa..Dd)=48'1.001hpps //  match-1: detelecine
 )                      //  miss-1
 (AaAbBcCcDd)           //   stride-2: 1.1..2.2..
 (Aa..Dd)=48'1.001hpps  //    match-2: detelecine
)                       //
pps]pfps                // weave + encode

Step 2. Next, convert NTSC-scan segments by adding SNME-decomb.

&="interpolate"                              // bind interpolate operator
30'1.001htfps[hpps                           // decode
(AaBbBcCdDd)                                 // stride-1: ..1.1..2.2
(                                            //
 (                                           //
  (Aa..Dd)=48'1.001hpps                      //  match-1: detelecine
 )                                           //  miss-1
 (AaAbBcCcDd)                                //   stride-2: 1.1..2.2..
 ((                                          //    change (match-2) to (((match-2))miss-2)
   (Aa..Dd)=48'1.001hpps                     //    match-2: detelecine
  )                                          //    miss-2
  (Ab)                                       //     stride-3: ..
  (AA&bA&$b(A^&bA&$b)A^&bA&bb)=120'1.001hpps //      match-3: SNME-decomb + weave + double pictures
 )                                           //
)                                            //
pps]pfps                                     // weave + encode

If the processing ended at this step, the 'NTSC'-hard segments would be encoded as 24'1.001pfps but the NTSC-scans would be encoded as 60'1.001pfps. A common rate must be found. The solution is 120'1.001pps.

Step 3. Next, add a picture rate fixup.

&="interpolate"
30'1.001htfps[hpps                                          // decode
(AaBbBcCdDd)                                                // stride-1: ..1.1..2.2
((                                                          //
  ((Aa..Dd)pps(A)((AA..&..&...&..&..$)AAAAA)=120'1.001pps)  //  match-1: detelecine + weave + 1-to-5 interpolation
 )                                                          //  miss-1
 (AaAbBcCcDd)                                               //   stride-2: 1.1..2.2..
 ((                                                         //
   ((Aa..Dd)pps(A)((AA..&..&...&..&..$)AAAAA)=120'1.001pps) //    match-2: detelecine + weave + 1-to-5 interpolation
  )                                                         //    miss-2
  (Ab)                                                      //     stride-3: ..
  (AA&bA&$b(A^&bA&$b)A^&bA&bb)pps(1)(11)=120'1.001pps       //      match-3: SNME-decomb + weave + double pictures
 )                                                          //
)                                                           //
]pfps                                                       // encode

There is a twist. Some of the telecined segments have bad pull-downs that would be nice to fix.


The bad pull-down appears to have been ordinary 8-to-10 telecine cinema that was subsequently converted back to 24'1.001htfps by dropping frame [B+b] then converted back to 30'1.001htfps once again by repeating frame [D+d].

What notation will fix the bad pull-downs? 
&="interpolate"
(AaBcCdDdDd)                // stride-1: .....12121 (bad 2-3-2-3 pull-down)
(AaBa&cBcCdDd)              //  match-1 [see note 1]
(AaBcCcCcDd)                // stride-2: ...12121.. (bad 3-2-3-2 pull-down)
(AbAbBcCcDd(A^&bAbBcCcDd))  //  match-2 [see note 2]

[note 1] synthesize a new 'b' halfpic by interpolating between 'a' & 'c' (i.e. b = a&c) and drop the extra 'Dd'.

[note 2] synthesize a new 'a' halfpic by interpolating between the prior 'd' & 'b' (i.e. a = ^&b) and drop the extra 'Cc'.


Step 4. The following notation puts it all together to make smooth video pictures of the whole "Making Of" documentary at 120pfps in a single transcode.

&="interpolate"
30'1.001htfps[hpps                            // decode
(AaBcCdDdDd)                                  // stride-1: .....12121 (bad 2-3-2-3 pull-down)
(AaBa&cBcCdDd)                                //  match-1
(AaBcCcCcDd)                                  // stride-2: ...12121.. (bad 3-2-3-2 pull-down)
(AbAbBcCcDd(A^&bAbBcCcDd))                    //  match-2
(AaBbBcCdDd)                                  // stride-3: ..1.1..2.2
(                                             //
 ((                                           //
   (Aa..Dd)pps(A)((AA..&..&...&..&..$)AAAAA)  //  match-3: detelecine + weave + 1-to-5 interpolation
 ))                                           //  miss-3

Extra credit.
Recall that:

1, 24'1.001pps to 30'1.001htfps via 8-to-10 telecine has 53% judder @ 6 jps — awful but people think it's unavoidable.

2, 24'1.001pps to 30'1.001htfps via 4-to-5 interpolation has 27% judder @ 6 jps — better, but not great.

3, 24'1.001pps to 120'1.001pfps via 1-to-5 interpolation has 15% judder @ 24 jps when viewed on a 60Hz TV — terrific but a time consuming transcode.

60Hz TVs that have variable rate screens will sync to and play 57.6'1.001pfps just fine. That presents an opportunity for improvement.
